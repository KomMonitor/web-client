Neuerungen in KomMonitor V4.0.0-beta.1

Version 4 von KomMonitor führt eine mandantenorientierte Weiterentwicklung des User- und Rechtekonzepts ein. Dadurch wird es möglich, in der gleichen KomMonitor Instanz mehrere voneinander unabhängige Mandanten anzulegen, die ihrerseits eigenständig User und Datenressourcen verwalten. Ressourcen und Organisationsstrukturen eines anderen Mandanten sind dabei nicht einsehbar.

Im Detail sind folgende Aspekte als wesentliche Änderungen zu beachten:
  • Anpassungen in der Admin-Ansicht
    • Mandanten und Untergruppenhierarchien Eine Organisationseinheit kann nun durch Superadministratoren als Mandant gekennzeichnet werden. Mandanten können ihrerseits eine beliebig tiefe Untergruppenhierarchie aufbauen, um User in verschiedene Untergruppen zuzuordnen und Datenfreigaben und -administrationen gezielt zu steuern.
    • User-Gruppenkonzept statt rollenbasierter Zugriffsschutz Seit Version 4 werden einzelne User in (Unter-)Gruppen eines KomMonitor Mandanten zugeordnet und erhalten so automatisch gruppenspezifische Administrationsaufgaben und Datensatzfreigaben. Für jede (Unter-)Gruppe kann dabei passgenau festgelegt werden, ob User dieser Gruppe Themen, User oder Datensätze im Namen der Gruppe administrieren dürfen. Ebenso gelten für alle User einer Gruppe gesetzte Lese- und Editierrechte auf assoziierte Datensätze.
    • Untergruppenverwaltung und Gruppenrechte in KomMonitor Das Verwalten von Mandanten und ihren Untergruppen sowie die Zuweisung administrativer Rechte (User-, Themen- und Datenverwaltung) sowie Lese- und Editierzugriffe auf bestimmte Datensätze pro (Unter-)Gruppe erfolgt vollständig über KomMonitor Eingabemasken.
    • vereinfachte Zuordnung zwischen Usern und Gruppen via Keycloak Um User in die (Unter-)Gruppen eines Mandanten zuzuordnen, nutzen Useradministratoren Eingabemasken der Open Source Software Keycloak, die für KomMonitor das Nutzer- und Rechtekonzept übernimmt. Seit Version 4 ist es ausreichend, User in eine oder mehrere Gruppen einzuordnen. Allein durch die Gruppenzugehörigkeit erhalten alle User einer Gruppe die gleichen Rechte. Die Rechte der Gruppe selbst wiederum werden vollständig in KomMonitor Menüs verwaltet.
    • neue explizite Eigentümerschaft an Datensätzen Für jede Datenressource (Raumeinheit, Indikator, Georessource) muss eine Eigentümer-Gruppe angegeben werden. Nur User der Eigentümer-Gruppe ist es gestattet, Datensatzfreigaben an andere Gruppen innerhalb des eigenen Mandanten zu kontrollieren sowie assoziierte Datensätze zu entfernen. Die Eigentümerschaft kann an andere Gruppen übertragen werden.
    • neue explizite Datensatz-Freigabe für die Öffentlichkeit Für jede Datenressource (Raumeinheit, Indikator, Georessource) wird die öffentliche Freigabe durch einen neuen expliziten Schalter festgelegt. Somit gibt es keine public Organisation mehr. Ist die öffentliche Freigabe für eine Ressource aktiviert, ist der Datensatz auch ohne Login abrufbar.
  • Anpassungen in der Kartenapp
    • Bugfixes Fehler in Reporting und Erreichbarkeitsanalysen wurden behoben.

// controls whether the mean values (arithmetic and regional) shall be displayed in legend and indicator statistics menus or not
window.__env.enableMeanDataDisplayInLegend = false;
								
// control additional average/mean lines in line chart, legend and statisticalValues 
// allowed options 
// 'both' --> both, arithmetic mean and regional mean will be displayed if available
// 'preferRegionalMeanIfAvailable'  --> only show one line at a time. regional mean will be preferred if available
// 'regionalMeanOrNone' --> never show arithmetic mean. show regional mean if available or nothing
window.__env.configMeanDataDisplay = 'preferRegionalMeanIfAvailable';
      Diese ermöglichen eine gezieltere Darstellung von regionalen Vergleichsmittelwerten und arithmetischem Mittel in Legende, Indikatoren-Statistiken und Zeitreihen- sowie Balkendiagrammen. Ist stattdessen window.__env.configMeanDataDisplay = 'both'; eingestellt, so werden in Legende, Zeitreihendiagramm und Indikatorenstatistik immer sowohl das arithmetische Mittel als auch ein optional eingetragener regionaler Vergleichswert angezeigt. Durch window.__env.configMeanDataDisplay = 'regionalMeanOrNone'; wird das arithmetische Mittel vollständig entfernt, sodass nur optional eingetragene regionale Vergleichswerte verwendet werden.

Neuerungen in KomMonitor V3.0.0

Version 3 von KomMonitor bietet eine in vielen Punkten überarbeitete und erweiterte Kartenapp sowie diverse neue Konfigurationsm&öuml;glichkeiten der integrierten Geo- und Sachdaten.
  • Anpassungen in der Admin-Ansicht
    • Klassifikation pro Indikator
      Für jeden Indikator kann eine individuelle Standard-Klassifizierung festgelegt werden, inklusive Anzahl Klassen, und Klassifikationsmethode. Auch k&öuml;nnen dabei pro Raumebene eigene Klassengrenzen vorgegeben werden.

      Jedem Indikator k&öuml;nnen gesamtregionale Vergleichswerte (Gesamtsumme, Durchschnittswert, räumlich nicht zuordenbare) im Metadaten-Bearbeiten Menü zugewiesen werden.     • Raumebenen als Umringslayer
      Pro Raumebene kann konfiguriert werden, ob sie in der Kartenapp als Umringslayer in der Layersteuerung erscheint.
      Dies erm&öuml;glicht es, die Grenzverläufe der Raumebene in die Karte als Orientierungshilfe einblenden zu lassen. Dies kann nützlich sein, um bspw. punkthafte Georessourcen optisch mit Raumebenen-Grenzen zu überblenden.
    • Textuelle POIs (punkthafte Georessourcen)
      Punkthafte Georessourcen erlauben auch statt eines Symbols eine textuelle Unterscheidung mit bis zu drei Zeichen.
    • aufgeräumte Farbpaletten in Klassifikation
      Die verfügbaren Farbpaletten des ColorBrewer wurden auf sequentielle beschränkt. Divergierende oder qualitative Paletten wurden entfernt.

      Für alle Geodatentypen wurde ein Fehler behoben, der das interaktive Editieren der Einzelzellen verhindert hat.
    • neue Konfigurationsparameter
      Zur Konfiguration neuer/erweiterter Funktionalitäten wurden neue Konfigurationsparameter ergänzt. Konkret:
      • Parameter zur Steuerung, ob im Säulendiagramm eine Durchschnittslinie sowie Namen der Raumeinheiten angezeigt werden sollen // show bar chart label and average line window.__env.showBarChartLabel = false; window.__env.showBarChartAverageLine = true;

Im Client muss hierzu die URL sowie der Prozessname gesetzt werden // KomMonitor Spatial Data Processor window.__env.targetUrlToSpatialDataProcessorInstance = 'http://localhost:8090/'; window.__env.spatialDataProcessor_processName_indicatorReachabilityStatistics = "isochrone-prune";
  • Anpassungen in der Kartenapp
    • NEU - erweiterte szenarienbasierte Erreichbarkeitsanalyse inkl. Indikatoren-Statistik-Verschneidung
      Die bishergie Erreichbarkeitsanalyse wurde stark ausgebaut zu einem Szenarienwerkzeug. Wesentliche Neuerungen:
      • Es k&öuml;nnen mehrere Erreichbarkeiten nacheinander angelegt und in der aktuellen Sitzung gespeichert werden.

Der Import von Adressdaten als punktbasierte Georessource mittels OpenStreetMap wurde dahingehend verbessert, dass eine Bewertung der möglichen Treffer durch den Backend-Dienst vorgenommen wird. Jeder erfolgreich importierte Punkt besitzt zwei Sachattribute, geocoderank und geocodedesc mit folgenden Wertemöglichkeiten:
      1. geocoderank=2 ‐ geocodedesc = high accuracy - low risk of false positioning. feature matches same street and city/postcode and same housenumber
      2. geocoderank=1 ‐ geocodedesc = moderate accuracy - medium risk of false positioning. feature matches same street and city/postcode. but housenumber does not match
      Somit können Mehrfachtreffer bestimmter Adressen direkt geokodiert werden. Wenn ein Ergebnistreffer der Suchanfrage gleiche Straße, Hausnummer und Ort/PLZ aufweist, ist es der bestmögliche Treffer (geocoderank=2). Stimmen immerhin Ort/PLZ und Straße überein, so kann zumindest eine ungenauere Koordinate ohne richtige Hausnummer bestimmt werden (geocoderank=1).

      Adressen, die nicht geocodiert werden können führen zu einer Fehlermeldung beim Import. Solche Einträge sollten auf Tippfehler geprüft werden. Sollte auch dann kein Treffer gefunden werden, so muss diese Adresse aus der Quelldatei entfernt werden und nachträglich manuell ergänzt werden.

Neuerungen in KomMonitor V2.0.0

Grundlegende Erweiterung des Datenmodells und des Rollen- und Rechtemanagements

Die Aktualisierung aller KomMonitor Komponenten auf eine Version >= V2.0.0 beinhaltet bedeutende Erweiterungen im Datenmodell sowie des gezielten Zugriffsschutzes auf einzelen Ressourcen. Version >= 2.0.0 umfasst folgende Anpassungen:

  • Einführung einer Ressource Organisation. Pro angelegter Organisation werden folgende vier Rollen/Rechtestufen unterschieden
    • <orgname>-viewer ‐ lesender Zugriff auf Datensätze
    • <orgname>-editor ‐ lesender + schreibender Zugriff auf Datensätze
    • <orgname>-publisher ‐ lesender + schreibender Zugriff auf Datensätze sowie Anlegen neuer Datensätze
    • <orgname>-creator ‐ lesender + schreibender + löschender Zugriff auf Datensätze sowie Anlegen neuer Datensätze
  • Die User Verwaltung erfolgt nun verpflichtend mit der freien Software Keycloak. Dort müssen weiterhin von KomMonitor Administratoren die User verwaltet werden und mit den passenden Organisationen und Rechtestufen assoziiert werden.
  • Alle Backend Dienste sind nun vollständig über Keycloak abgesichert, sodass nur noch autorisierte Benutzer systemrelevante Aktionen wie das Antriggern von Indikatorenberechnungen oder Editieren von Konfigurationsdateien durchführen können. Dazu müssen Anpassungen in der Keycloak-Umgebung vorgenommen werden. Sprechen Sie für Unterstützung Christian Danowski-Buhren von der Hochschule Bochum an.
  • Anpassungen in der Admin-Ansicht
    • erweiterte Verwaltung von Organisationen und Rollen/Rechtestufen
    • Editierbarkeit von Raumeinheits-Tabellen. Für Indikatoren, Raumebenen und Georessourcen über das Raumeinheit fortführen-Menü lassen sich die einzelnen Datenzeilen in einer Raumeinheits-Tabelle anzeigen. Ein Doppelklick auf eine editierbare Zelle ermöglicht dem User nun, den Zellenwert zu ändern und diese Änderung in der Datenbank dauerhaft zu übernehmen.
    • Editierbarkeit von Georessourcen-Raumeinheiten. Für jeden Georessourcen Datensatz bietet das Raumeinheit fortführen-Menü eine neue Möglichkeit, Einzelobjekte hinzuzufügen (bspw. einen neuen Standort einer neuen Kita manuell hinzufügen). Dazu kann die Geometrie des Objekts mit Hilfe eines Kartenwerkzeugs erstellt werden und mittels Textfeldern können die Sachattribute des neuen Objekts gesetzt werden.
    • Geocoding-Prototyp beim Import von Adressdaten als punktbasierte Georessource. Ein neuer Backend-Dienst wurde entwickelt, der auf Basis von OpenStreetMap Daten ein Geocoding von Adressdaten umsetzt. In der Admin-Ansicht kann nun beim Import von punktbasierten Georessourcen eine Excel- oder CSV-Datei mit Adressdaten angegeben werden, sodass erfolgreich geocodierbare Objekte als POI in die KomMonitor Datenbank übernommen werden können. Das Resultat ist stark abhängig von der Qualität der in OpenStreetMap hinterlegten Daten. Der Geocodierungsdienst wird von der Hochschule Bochum betrieben und basiert auf der Software Nominatim. In künftigen Updates wird diese Mechanik weiter verbessert.
  • Anpassungen in der Kartenapp
    • Überarbeitete Reporting-Funktion in der Kartenapp. Es lassen sich nun Reporting Vorlagen auswählen, die dann mit gewünschten Metadaten und Indikatorendaten gefüllt werden können (Zeitpunkt und zeitliche Entwicklung). Außerdem können Isochronen um punktbasierte Georessourcen in einem Report erstellt werden.
  • diverse Bugfixes und Aktualisierungen einiger Programmbibliotheken.
